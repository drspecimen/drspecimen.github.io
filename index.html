03 — Technical Notes

Build logs & write-ups

Detailed write-ups of selected builds — architecture decisions, problems encountered, and how they were solved.

The problem

Keystone Bank's data centre infrastructure — UPS arrays, CRAC cooling units, generators — was tracked across a combination of paper logs, shared Excel files, and individual engineer memory. There was no single view of system status, no threshold alerting, and no incident history. The cost was time: engineers spent part of every shift manually checking and recording readings that could be centralised.

Architecture decision: why NW.js

A web app was the obvious first instinct. The problem: the data centre environment has strict network controls and no guaranteed internet access for internal tools. NW.js (formerly Node-WebKit) gave us a Chromium rendering engine plus full Node.js access in a single desktop binary — no browser needed, no server to host, no network dependency. The UI is built in plain HTML/CSS/JS, which meant rapid iteration, and Node handles the file system and any local data persistence.

What it does

  • Real-time status dashboard across UPS, CRAC, and generator systems
  • Configurable threshold alerts — visual and logged — when readings exceed set parameters
  • Incident history log with timestamps and engineer notes
  • Export-ready shift reports replacing the manual log sheets

Result

The tool is in production use. It replaced the fragmented tracking system and reduced the time spent on routine status checks. The next planned evolution is a LAN multi-user architecture so that all engineers on a shift share a single live view without needing a central server.

Scope

The DIG Courses module adds a full learning management system to the DIG Platform — tutor registration with admin approval, a drag-and-drop course builder, student enrollment with Paystack payment handling, lesson progress tracking, assignment submissions, discussion forums per course, and certificate generation as a signed PDF.

The enrollment isolation problem

An early version of the system used a browser cookie keyed only by course_id to track enrollment. This meant that if two students used the same browser on the same device, the second student would see the first student's progress, completion status, and certificate — including the wrong name on the certificate. The fix was to remove cookie-based enrollment detection entirely from all page-render files and scope every lookup to the logged-in student's session email: WHERE course_id = :cid AND student_email = :email.

Certificate generation

Certificates are generated as A4 landscape PDFs using DOMPDF. The template embeds: the student's name, course title, issue date, a unique credential ID, the tutor's drawn signature (captured as a base64 PNG from a canvas pad in the tutor dashboard), and the DIG founder signature. All images are embedded as base64 data URIs so DOMPDF doesn't need HTTP access. The PHP GD extension is required — a dependency that's easy to miss on fresh XAMPP setups.

Commission tracking

When a paid course sells, the platform records the gross amount, the DIG platform fee (configurable in admin settings), and the tutor's cut in a tutor_earnings table. Admins can review per-tutor balances, see bank details, and mark payouts as completed. A key bug fixed during build: getSetting() returns an empty string for missing keys, not null, so using ?? instead of ?: caused the commission rate to silently evaluate to 0%.
