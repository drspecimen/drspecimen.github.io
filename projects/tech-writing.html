NETWORK MONITOR (INFRADASH)

Troubleshooting a recurring fault on a Socomec Delphys Green 2.0 UPS where its network management card would silently lose its web interface every few hours, while still responding to ping, with the only recovery path being a manual Telnet session to force-restart the agent. Diagnosing that single device exposed a much larger need, existence of a single place to see the health of all connected infrastructure at once and a historical record of downtime to support escalation with vendors. Some Organizations running physical infrastructure, including data centers, and server rooms, typically rely on a patchwork of web interfaces with no centralized visibility.

This creates several recurring operational issues:

  • No single dashboard to see which devices are online, degraded, or offline at a glance,
  • No historical uptime data to support root-cause analysis or vendor escalation,
  • Browser-based dashboard approaches are fragile in practice, cross-origin restrictions and iframe security policies block full navigation into vendor login pages and multi-step interfaces, which means an embedded dashboard often cannot actually drive the underlying device, and
  • No lightweight, low-cost way to deploy a monitoring layer without committing to expensive enterprise DCIM platforms

That need became the starting point for InfraDash.

Scoped around fouz core objectives:

  1. Build a single desktop application that consolidates access to multiple infrastructure web interfaces, including UPS, CCTV, and environmental sensors, with the ability to extend to any future IP-based system.
  2. Provide continuous uptime monitoring with historical visualisation, not just a point-in-time status check.
  3. Guarantee full, unrestricted browser-grade navigation into each device's native web interface, solving the iframe and cross-origin limitations that break most lightweight dashboard approaches.

Design and Architecture

The first prototype used HTML iframes to embed device interfaces inside a browser dashboard. This failed quickly and instructively: most embedded systems enforce frame-blocking policies and block iframe navigation outright, meaning a login page might load but subsequent navigation, such as clicking into a UPS configuration menu or following a CCTV login redirect, simply would not work. This was the key design lesson of the project: a monitoring dashboard for real hardware needs a real browser engine per device, not a sandboxed iframe. That led to a deliberate architectural pivot toward a desktop application built on NW.js, using a native embedded browser tag that runs a full, isolated Chromium instance per connected device. This single decision solved the navigation problem completely. CCTV redirects, multi-step UPS configuration menus, and authentication flows now behave exactly as they would in a standalone browser tab, regardless of vendor.

Core components implemented:

  • Tabbed device sessions, where each device opens in its own persistent tab with full page state preserved and instantly switchable,
  • Live ping-based health monitoring, with a background process polling every registered device at a configurable interval and logging uptime and downtime events with timestamps,
  • A dashboard analytics layer showing aggregate stats such as online and offline counts, average uptime, category breakdown, a 24-hour uptime visualisation per device, and a recent events feed,
  • An auto-rotate display mode that cycles through all open device tabs at a configurable interval, designed for use on a wall-mounted operations-style monitor,
  • A full-view kiosk mode with an integrated screen wake lock, which removes all interface chrome and prevents the display from sleeping, allowing the dashboard to run unattended on a dedicated screen,
  • A built-in CLI terminal panel, styled after a code editor's integrated terminal, offering direct ping, traceroute, ipconfig, and arbitrary command access per device,
  • A white-label branding system allowing organisation name, window title, and logo to be fully edited from within the application, including a live-updating taskbar and window icon, with no source code access required, alongside a factory reset option to return the application to a clean default state, and
  • Persistent local storage for device lists, uptime history, and configuration, with export and import support for backup or migration between machines.

Implementation

The application is built entirely in HTML, CSS, and JavaScript, running inside NW.js, which provides direct Node.js access from within the application itself. This enables native operating system interaction, including process spawning for the terminal, system calls for the screen wake lock, and file system access for persistent configuration, without requiring a separate backend server. The entire stack is self-contained in a single distributable folder, with no external dependencies or internet access required at runtime. For distribution, the application is packaged using Inno Setup into a standard Windows installer, allowing staff to install it like any commercial desktop application, complete with Start Menu and desktop shortcuts.

Deployment

The tool is currently deployed for internal infrastructure monitoring. The architecture is intentionally general-purpose: any IP-addressable device with a web interface, including IoT devices, network switches, routers, firewalls, printers, and access control systems, can be added through the interface without any code changes, making it suitable for branch network monitoring as well as data centre and server room use cases.

Scope

The DIG Courses module adds a full learning management system to the DIG Platform — tutor registration with admin approval, a drag-and-drop course builder, student enrollment with Paystack payment handling, lesson progress tracking, assignment submissions, discussion forums per course, and certificate generation as a signed PDF.

The enrollment isolation problem

An early version of the system used a browser cookie keyed only by course_id to track enrollment. This meant that if two students used the same browser on the same device, the second student would see the first student's progress, completion status, and certificate — including the wrong name on the certificate. The fix was to remove cookie-based enrollment detection entirely from all page-render files and scope every lookup to the logged-in student's session email: WHERE course_id = :cid AND student_email = :email.

Certificate generation

Certificates are generated as A4 landscape PDFs using DOMPDF. The template embeds: the student's name, course title, issue date, a unique credential ID, the tutor's drawn signature (captured as a base64 PNG from a canvas pad in the tutor dashboard), and the DIG founder signature. All images are embedded as base64 data URIs so DOMPDF doesn't need HTTP access. The PHP GD extension is required — a dependency that's easy to miss on fresh XAMPP setups.

Commission tracking

When a paid course sells, the platform records the gross amount, the DIG platform fee (configurable in admin settings), and the tutor's cut in a tutor_earnings table. Admins can review per-tutor balances, see bank details, and mark payouts as completed. A key bug fixed during build: getSetting() returns an empty string for missing keys, not null, so using ?? instead of ?: caused the commission rate to silently evaluate to 0%.
